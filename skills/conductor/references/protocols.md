# Conductor Protocols & Schemas

## progress.json Schema

```json
{
  "plan_file": "path/to/plan.md",
  "started_at": "2026-02-27T10:00:00Z",
  "max_workers": 2,
  "stages": {
    "0": {
      "id": "0",
      "title": "Environment Setup",
      "status": "pending",
      "dependencies": [],
      "spec_file": ".claude/conductor/stages/0.md",
      "log_file": ".claude/conductor/logs/0.log",
      "retry_count": 0,
      "started_at": null,
      "completed_at": null,
      "files_created": [],
      "files_modified": [],
      "error": null,
      "notes": null,
      "red_team_issues": null
    }
  },
  "skipped": [],
  "chain_depth": 0,
  "completed_at": null
}
```

### Status Values

```
pending → red-teaming → running → verifying → complete
                     ↘ flagged → running (user proceeds)
                                → pending (user revises spec)
                        running ↘ failed → pending (retry)
                                          → fatal (3 retries exhausted)
blocked (dependency is fatal)
skipped (user --skip flag)
```

- `pending`: waiting for dependencies or dispatch
- `red-teaming`: red team reviewer is evaluating the stage spec
- `flagged`: red team found issues, awaiting user decision
- `running`: worker subagent is executing
- `verifying`: verifier subagent is checking results
- `complete`: stage done, verified
- `failed`: worker reported failure (will retry)
- `fatal`: 3 retries exhausted, cannot proceed
- `blocked`: a dependency is fatal, this stage cannot run
- `skipped`: user explicitly skipped via --skip flag

## Stage Spec Format

Each stage spec is written to `.claude/conductor/stages/{id}.md`:

```markdown
# Stage {id}: {title}

## Dependencies
- Stage {dep_id}: {dep_title} (status: {status})

## Existing Context
<!-- Auto-generated from code map. Gives the worker orientation about existing files. -->
<!-- Omitted when --no-codemap is set or code map generation fails. -->

### Relevant Files
| File | Purpose | Key Types/Functions |
|------|---------|-------------------|
| path/to/file.py | Brief description of what this file does | ClassA, function_b(), CONSTANT_C |
| path/to/other.py | Brief description | ClassD, function_e() |

### Code Map Summary
{filtered summary relevant to this stage's deliverables and plan text}

## Deliverables
- CREATE: `path/to/new/file.py`
- MODIFY: `path/to/existing/file.py`

## Test Criteria
- Description of what to test
- Specific commands to run
- Expected outcomes

## Plan Text
{verbatim text from the original plan for this stage}

## Context from Prior Stages
{populated at dispatch time with results from completed dependencies}
```

### Existing Context — How It Works

The "Existing Context" section is auto-generated by the conductor using the [code map generator](/home/alfonso/Projects/code-map-generator/) before stage specs are created. It provides workers with file-level orientation so they spend fewer turns on exploration.

**Generation process:**
1. The conductor runs `bash /home/alfonso/Projects/code-map-generator/run.sh <project_path>` once to generate/update the full code map.
2. For each stage, the conductor runs `bash /home/alfonso/Projects/code-map-generator/run.sh <project_path> --select "<stage description>" --json` to get files relevant to that stage.
3. The JSON output is parsed and formatted into the "Relevant Files" table and "Code Map Summary".

**When omitted:**
- The `--no-codemap` flag is set on the conductor invocation
- The code map generator is unavailable or fails
- The auto-context selection returns no relevant files for a stage

When omitted, the stage spec is created without the "Existing Context" section and workers discover the codebase through their own exploration — the same behavior as before this feature was added.

## Log Format

Worker logs are written to `.claude/conductor/logs/{id}.log`.

Each line follows this format:
```
[ISO8601] STATUS: message
```

### Log Statuses

| Status | Meaning |
|--------|---------|
| `STARTED` | Worker began execution |
| `PLANNING` | Worker is analyzing the stage spec and planning approach |
| `CREATING` | Worker is about to create/modify a file (write-ahead) |
| `CREATED` | File creation/modification complete |
| `RUNNING` | Worker is running a command (e.g., installing deps) |
| `RESULT` | Command result (stdout/stderr summary) |
| `TESTING` | Worker is running test criteria |
| `TEST_PASS` | A test criterion passed |
| `TEST_FAIL` | A test criterion failed |
| `ERROR` | An error occurred (with details) |
| `COMPLETE` | Stage completed successfully |
| `FAILED` | Stage failed (with reason) |

### Example Log

```
[2026-02-27T10:00:01Z] STARTED: Stage 1a — State Collector
[2026-02-27T10:00:02Z] PLANNING: Will create brainstem/collector.py with psutil-based system metrics collection

[2026-02-27T10:00:03Z] CREATED: brainstem/__init__.py
[2026-02-27T10:00:04Z] CREATING: brainstem/collector.py
[2026-02-27T10:00:15Z] CREATED: brainstem/collector.py (45 lines)
```

```
You are a build-worker. Read and follow the worker protocol.

## Your Assignment
Stage: {id} — {title}

## Stage Spec
{full contents of .claude/conductor/stages/{id}.md}

## Log File
Write your progress to: {project_root}/.claude/conductor/logs/{id}.log
Format: [ISO8601] STATUS: message
Log BEFORE and AFTER every file operation (write-ahead logging).

## Context from Prior Stages
{for each completed dependency:
  - Stage {dep_id}: {dep_title}
    Files created: {list}
    Files modified: {list}
    Notes: {notes}
}

## Retry Context
{if retry_count > 0:
  This is retry #{retry_count}. Previous attempt failed:
  Error: {error from progress.json}
  Previous log: {contents of log file}
  Fix the issue and try again. Do not repeat the same mistake.
}

## Project Instructions
```


## Red Team Prompt Template

```
You are a red-team reviewer. Your job is to catch conceptual errors in a stage spec BEFORE any code is written.

## Stage to Review
Stage: {id} — {title}

## Stage Spec
{full contents of .claude/conductor/stages/{id}.md}

## Context from Prior Stages
{for each completed dependency:
  - Stage {dep_id}: {dep_title}
    Files created: {list}
    Notes: {notes}
}

## Your Review Criteria

Evaluate the stage spec for:
1. **Contradictions**: Does the approach contradict itself or conflict with prior stage outputs?
2. **Feasibility**: Are the deliverables achievable given the dependencies and context?
3. **Missing considerations**: Are there obvious gaps — missing files, unhandled edge cases, wrong integration points?
4. **Wrong abstractions**: Does the proposed structure make sense, or is it over/under-engineered for the task?
5. **Dependency issues**: Does this stage assume something from a dependency that wasn't actually delivered?

## Rules
- Review the SPEC only. Do not write code, do not create files, do not run commands.
- Be concise. This is a fast gate, not a design review.
- Only flag issues that would likely cause the worker to fail or produce wrong results.
- Do NOT flag style preferences, minor naming choices, or speculative concerns.
- If the spec looks sound, say PASS and move on.

## Return Format
Return EXACTLY one of these structured results as your final message:

RED_TEAM: PASS
SUMMARY: No blocking issues found.

— or —

RED_TEAM: FLAG
ISSUES:
- {concrete issue 1}
- {concrete issue 2}
SUMMARY: {one-line summary of why this needs attention}
```

## Arbiter Prompt Template

```
You are an arbiter. A red-team reviewer flagged issues with a stage spec. Your job is to decide what to do about it — WITHOUT escalating to a human unless absolutely necessary.

## Stage Under Review
Stage: {id} — {title}

## Stage Spec
{full contents of .claude/conductor/stages/{id}.md}

## Red Team Issues
{issues from the red team FLAG result}

## Original Plan
{full contents of the plan file — this is the source of truth for the platform's overall intent}

## Context from Prior Stages
{for each completed dependency:
  - Stage {dep_id}: {dep_title}
    Files created: {list}
    Notes: {notes}
}

## Your Decision Framework

You have three options. Choose the LEAST escalatory one that resolves the problem:

### Option 1: SYNTHESIZE (preferred)
The red team issues are valid but fixable. You can revise the stage spec to address them without changing what the platform is supposed to do. Produce a corrected stage spec that resolves the flagged issues while staying true to the original plan's intent.

### Option 2: PROCEED
The red team issues are technically valid but non-blocking — the worker will likely succeed despite them, or the issues are speculative enough that they don't warrant delaying the stage.

### Option 3: ESCALATE (last resort)
Use ONLY when:
- There is no workable synthesis — the issues reveal a fundamental contradiction that cannot be resolved by revising this stage alone (e.g., it requires changes to already-completed stages or to the dependency graph)
- The resolution would fundamentally change the platform's intent as described in the original plan — not just this task's approach, but what the product is supposed to DO

Do NOT escalate for:
- Disagreements about implementation approach (synthesize instead)
- Missing edge case handling (synthesize or proceed)
- Suboptimal but functional designs (proceed)

## Return Format
Return EXACTLY one of these as your final message:

ARBITER: SYNTHESIZE
REASONING: {why the issues are valid and how you resolved them}
REVISED_SPEC:
{complete revised stage spec in markdown — this will replace the existing spec file entirely}

— or —

ARBITER: PROCEED
REASONING: {why the flagged issues are non-blocking}

— or —

ARBITER: ESCALATE
REASONING: {why this cannot be resolved without human input}
SEVERITY: {NO_SYNTHESIS | INTENT_CHANGE}
```

## Structured Result Formats

### Worker Success Result
```
RESULT: SUCCESS
FILES_CREATED: path1.py, path2.py
FILES_MODIFIED: path3.py
TESTS_PASSED: test description 1, test description 2
NOTES: any relevant observations or decisions made
```

### Worker Failure Result
```
RESULT: FAILURE
ERROR: description of what went wrong
FILES_CREATED: any files created before failure
FILES_MODIFIED: any files modified before failure
ATTEMPTED_FIX: description of fix attempts
```

### Verifier Pass Result
```
VERIFICATION: PASS
CHECKS_RUN: 5
CHECKS_PASSED: 5
ISSUES: none
```

### Verifier Fail Result
```
VERIFICATION: FAIL
CHECKS_RUN: 5
CHECKS_PASSED: 3
ISSUES:
- CRITICAL: brainstem/collector.py missing collect() function
- WARNING: No docstring on main() function
- INFO: Unused import in line 3
```

### Red Team Pass Result
```
RED_TEAM: PASS
SUMMARY: No blocking issues found.
```

### Red Team Flag Result
```
RED_TEAM: FLAG
ISSUES:
- Stage assumes config.yaml exists but Stage 0 only creates config.json
- Deliverable says MODIFY auth.py but no prior stage creates it
SUMMARY: File format mismatch and missing dependency output
```

### Arbiter Synthesize Result
```
ARBITER: SYNTHESIZE
REASONING: Red team correctly identified config format mismatch. Revised spec to use config.json instead of config.yaml, matching Stage 0 output.
REVISED_SPEC:
# Stage 1a: Config Loader
...
```

### Arbiter Proceed Result
```
ARBITER: PROCEED
REASONING: The flagged edge case (empty config) is handled by the framework's default config mechanism and will not cause worker failure.
```

### Arbiter Escalate Result
```
ARBITER: ESCALATE
REASONING: This stage's deliverables require a REST API, but the original plan describes a CLI tool. Resolving this would change what the product does, not just how it's built.
SEVERITY: INTENT_CHANGE
```

## Concurrent Safety Rules

1. **Max workers**: default 2, configurable via `--max-workers`. Stages in `red-teaming` count toward this cap.
2. **No overlapping file writes**: dependency graph ensures stages that touch the same files run sequentially
3. **Supervisor is sole writer of progress.json**: workers never read or write it
4. **Workers write only to**: their assigned deliverable files + their own log file
5. **Verifiers are read-only**: they cannot create or modify any files
6. **Log files are append-only**: one writer per log file (the assigned worker)
7. **Red team agents are read-only**: they cannot create, modify, or delete any files
8. **Arbiter agents are read-only**: they return revised specs as text in their result; the conductor writes the file
